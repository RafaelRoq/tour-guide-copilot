# JSON Schema — Intermediate Format

This document defines the JSON structure that sits between the AI
generation step and the rendering step. The AI produces this JSON
from the guide's document. The renderer consumes it to generate
HTML, markdown, or other formats.

## Why a schema matters

The schema is the contract between two independent parts of the
system. Anyone can write a new renderer (PDF, Telegram bot, native
app) without touching the generation logic, as long as they read
this format. And anyone can improve the prompts without breaking
existing renderers, as long as the output still conforms to this
schema.

## Field reference

### `guide`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | yes | Guide's display name |
| `city` | string | yes | City of the tour |
| `language` | string | yes | ISO 639-1 code (`en`, `es`, `fr`...) |
| `tagline` | string | no | One-line description in the guide's voice. Generated by AI from the document. If the guide doesn't provide enough personality, omit. |

### `itinerary`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `title` | string | yes | Name of the tour |
| `duration_minutes` | integer | no | Estimated duration. Only if the guide mentions it. |
| `stops` | array | yes | Ordered list of stops (see below) |

### `itinerary.stops[]`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `order` | integer | yes | Position in the route (1-based) |
| `name` | string | yes | Name of the stop |
| `description` | string | yes | 1-3 sentences. What makes this stop interesting, in the guide's voice. |
| `tip` | string | no | Practical tip specific to this stop (best time, photo spot, what to notice). Null if the guide didn't mention one. |
| `coordinates` | object | no | `{ "lat": float, "lng": float }`. Extracted if the guide provides an address or well-known landmark. Null if ambiguous. |

### `plans[]`

Each plan is a **situation** — a question the tourist might ask after
the tour. The AI generates these from the guide's document. If the
guide didn't cover a topic, the plan has an empty `recommendations`
array (never invented content).

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `situation` | string | yes | The question, phrased as the tourist would ask it |
| `slug` | string | yes | URL-friendly identifier (for anchors and routing) |
| `recommendations` | array | yes | List of recommendations (see below). Empty array if the guide has no info. |

### `plans[].recommendations[]`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | yes | Name or description of the place/activity |
| `description` | string | yes | The guide's personal take — why this place, what's the story. 1-2 sentences in the guide's voice. Never null. |
| `what_to_order` | string | no | What to get there — the guide's specific pick |
| `what_to_avoid` | string | no | What to skip or watch out for |
| `price_range` | string | no | Free-text indicator: `"Free"`, `"€"`, `"€€"`, `"€€€"` |
| `vibe` | string | no | Short description of the atmosphere |
| `coordinates` | object | no | `{ "lat": float, "lng": float }` |

### `warnings`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| (array of strings) | string[] | no | General warnings that don't belong to a specific plan. Tourist traps, scams, transport tips. |

### `metadata`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `generated_at` | string | yes | ISO 8601 timestamp |
| `generator_version` | string | yes | Semver of the tool that generated this |
| `model` | string | yes | Model used for generation |
| `source_file` | string | yes | Filename of the guide's input document |

## Design principles

**1. Never invent.**
If a field has no information from the guide, it's `null` or an empty
array — never fabricated. The `recommendations` array being empty is a
valid and expected state.

**2. Guide's voice, not AI's voice.**
The `description`, `tip`, `what_to_order`, and `what_to_avoid` fields
should preserve the guide's tone. "Skip the paella, get the arroz"
is better than "The arroz a banda is recommended over the paella."

**3. Situations, not categories.**
Plans are organized by questions the tourist actually asks ("Where do
I eat after the tour?"), not by abstract categories ("Restaurants").
This is a deliberate design choice — it maps to how people think when
they're standing in a city with free time.

**4. Coordinates are optional everywhere.**
Not every guide will provide addresses. Not every place is a
well-known landmark the AI can geocode confidently. When in doubt,
omit coordinates rather than guess.

**5. Flat and simple.**
No deeply nested structures. Any renderer should be able to iterate
over `stops` and `plans` with a simple loop. This keeps the barrier
to contribution low.

## Standard situations

These are the default situations the AI will try to generate plans for.
If the guide's document has relevant knowledge, the situation gets
populated. If not, it gets an empty `recommendations` array.

| Slug | Situation |
|------|-----------|
| `eat-after-tour` | Where do I eat after the tour? |
| `2-free-hours` | What do I do with 2 free hours? |
| `bar-tonight` | A good bar tonight? |
| `breakfast-tomorrow` | Where should I have breakfast tomorrow? |
| `rainy-day` | What if it rains? |
| `local-shopping` | Where do locals actually shop? |

Guides can add more situations through their document. The AI will
detect them and create new plan entries. These six are starting points,
not a closed list.

---

## Extending the schema

If a contributor wants to add a new top-level section (e.g., `audio`
for future TTS, or `reviews` for tourist feedback), they should:

1. Add it as an optional top-level key.
2. Document it in this file.
3. Ensure all existing renderers ignore unknown keys gracefully.

The schema should grow through optional additions, never through
breaking changes to existing fields.
